## 第2回 ソフトウェア工学概論（1）

### ソフトウェアの定義
1. 実行されることによって必要な特性・機能・性能を提供する命令語群であるコンピュータプログラム
2. プログラムが適切に情報を扱うことを可能とするデータ構造
3. プログラムの操作や使用法を記述した情報であるドキュメント
> ハードウェアと異なり物理的な劣化はしないが、環境の変化に伴う設計の悪化は起こる。

### ソフトウェアの4つの進化特性
- **環境適応**：新しい実行環境や技術的なニーズを満たすよう適応し続けなければならない。
- **機能強化**：新しいビジネス要求やユーザーの利便性を実現するために強化しなければならない。
- **システム拡張**：近代的な外部システムやデータベースと相互運用できるよう拡張しなければならない。
- **再構築**：進化し続けるインフラ環境で実行可能となるよう、内部構造を再構築しなければならない。

### ソフトウェア工学の定義とゴール
- **IEEEによる定義**：
  > 「ソフトウェアの開発、運用、メンテナンスに対するシステマティックで規律ある、定量化できるアプローチの適用」
- **最終的なゴール**：品質、コスト、納期の最適なバランスであるQCDの最適化を実現するための手法や方法論の確立。

### Hookerの7原則
1. **ユーザーへの価値提供**：システムが存在する唯一の理由。
2. **シンプルさの維持**：KISS原則として知られる通り、設計は極力シンプルに保つ。
3. **明確なビジョンの維持**：一貫したビジョンがプロジェクトの成功に不可欠である。
4. **他者への配慮**：自分の作ったものを他人が理解できるよう、設計や実装、文書化を行う。
5. **変化への開放性**：未来の変化や拡張に対応できる汎用的なシステムを構築する。
6. **再利用の計画性**：コンポーネントを再利用可能に設計し、将来のコストを低減する。
7. **事前の思考**：行動を起こす前に、簡潔明瞭な考えを持つことが最良の結果を生む。

### ソフトウェア工学が必要とされる4つの背景
1. **大規模化と複雑さの増大**：社会インフラとしてソフトウェアへの依存度が高まり、効率と品質の両立が必須となった。
2. **IT人材の不足**：量的および質的な人材不足が深刻化しており、課題解決型や価値創造型の人材が求められている。
3. **社会的使命の重み**：ライフライン等に組み込まれるシステムが増え、極めて高い信頼性が要求される。
4. **不確実性の増大**：検索型から対話や交流型への変化など、意図しない形での活用やニーズの多様化が進んでいる。

### プログラミングとソフトウェア工学の違い
- **時間**：単に動くだけでなく、将来の依存ライブラリやOS、ハードウェアの更新に耐えうる持続可能性を検討する。
- **スケール**：コードベースの肥大化や、開発および保守に関わる人員数の拡大に対応する。
- **トレードオフ**：限られたリソースの中で、時間とスケールの最適解を導き出す。
> **スケーラブルなソフトウェア**：欠陥の修正コストは、着想から設計、開発、テストを経て本番環境へと工程が進むほど指数関数的に増大する。そのため、存続期間を通して安全に変更可能であるという持続可能性が重視される。

## 第3回 ソフトウェアライフサイクル

### ソフトウェアライフサイクルの3段階
1. **誕生**：ニーズの発生から企画や計画を経て、要件定義書を作成する。
2. **開発・運用**：設計やコーディングを行い、実稼働から保守運用へと進む。
3. **廃止**：サービスの終了、または新規ソフトウェアへのリニューアルを行う。

### 要件定義の概要と意義
- **主な記載項目**：システムの目的や概要、機能、システム構成図、目標性能、他システム連携仕様、運用制限事項、開発スケジュールや体制、納品物。
- **要件定義の意義**：
  - 実現する機能と実現しない機能を明確にするフィージビリティスタディとしての役割を持つ。
  - プロジェクト関係者間で合意範囲であるスコープを共有するためのバイブルとする。
  - 外部委託時の提案依頼書であるRFPの基礎資料として活用する。

### 設計・制作と外部調達
- **仕様化**：要件定義書の内容を、実際のコーディングが可能なレベルの設計書や仕様書に落とし込む。
- **体制づくり**：自社開発である内製か、外部委託である調達かを判断し、作業分解図であるWBSを制作する。
- **オフショア開発**：コスト削減を目的に海外企業へ外部委託する手法。文化や言語、商習慣、品質意識の違いによるトラブルに注意が必要である。

### テストとデバッグ
- **基本前提**：バグのないソフトウェアは存在しないという前提に立って設計および開発を行う。
- **早期発見の重要性**：あらかじめテスト計画を立て、バグを早期に発見して対処することがコスト抑制につながる。
- **テストの限界**：工数と時間の制約上、すべてのパターンをテストすることは不可能である。そのため、どの箇所をどの程度テストするかという戦略が重要になる。

### 運用・保守とドキュメンテーション
- **プロアクティブな運用**：稼働後に得られる各種データを能動的に活用し、システムの改善につなげる。
- **バスタブ曲線**：初期故障期の高発生率から安定稼働期の低減を経て摩耗故障期の再上昇に至るという故障率の変化特性を考慮した保守計画が必要である。
- **文書化**：複数人での開発において、非同期コミュニケーションを円滑にするためにドキュメンテーションが必須となる。国際プロジェクトを想定し、基本は英語での記述が標準となる。

## 第4回 プロジェクトマネジメント（1）

### プロジェクトの2大特性
- **有期性**：明確な始まりと必ず終わりというゴールがある類の仕事であり、定常業務やルーティンワークとは異なる。
- **独自性**：独自の目的や、これまでにない新しい成果物を達成する類の仕事。

### フォアキャスティングとバックキャスティングの比較
- **フォアキャスティングの特徴**：
  - どれだけやったかというインプットの視点を重視する。
  - 現在の立ち位置や今あるリソースから順に積み上げるように進める。
  - 具体的な例として学校の定期試験の勉強などが挙げられるが、予期せぬ変化によって目標に到達できないリスクがある。
- **バックキャスティングの特徴**：
  - 何を成し遂げたかというアウトプットの視点を重視する。
  - 将来の目標から逆算して、今何が必要であるかを計画しながら進める。
  - ビジネスにおけるプロジェクト業務の進め方がこれに該当し、初期段階での明確な目標設定や定義が必要となる。
> プロジェクトマネジメントにおいては、常に逆算型であるバックキャスティングの視点で計画および進行することが求められる。

### QCDによるソフトウェア分析
- **QCDの概念**：品質、コスト、納期の3軸。
- **システムの性質による優先度の変化**：
  - **品質優先**：医療機器や自動運転など、人命や社会インフラに直結するシステム。
  - **費用優先**：予算の予算枠が決まっている公共関連システムや自治体システム。
  - **納期優先**：特定のイベントや行事、あるいは法改正の施行日に合わせる必要があるシステム。

### ソフトウェアの4つの評価指標・規模見積もり
1. **ソースコード行数**：総ステップ数とも呼ばれ、プログラムの物理的な量で測る指標。
2. **オブジェクト容量**：コンパイル後のデータサイズ。組み込み型ソフトウェアなどでは、サイズが小さいほど評価が高くなる。
3. **ファンクションポイント法**：プログラムを書く前に、実装する機能の数や複雑さからシステム規模を客観的に概算する手法であり、FP法と略される。
   - $FP値 = 基準値 \times (0.65 + \frac{調整値}{100})$
   - $原価 = FP値 \times 作業単価 \times 換算値$
4. **使い勝手**：直感性を評価するもので、画面の視認性、操作性、入力補助、互換性、ガイダンスの充実度を指す。

### 品質管理とバグ発生率
- **品質指標**：1ステップあるいは1行あたりに発生したバグの数で表す。
- **バグ発生率の捉え方**：テストを充実させるほど発見バグ数が増えるため、一時的に発生率は高くなる。これは品質が悪いわけではなく、テストが十全に行われたことによる品質向上を意味する場合がある。
- **Hyrumの法則**：インターフェースに十分な数のユーザーがいる場合、仕様が何を保証しているかは関係なく、システムのあらゆる観察可能な挙動に依存したユーザーが生まれるという法則。バグがないことの完全な証明は困難であるため、この法則を踏まえた堅牢な設計が重要となる。

## 第5回 WBS（Work Breakdown Structure）

### WBSの定義
> 「プロジェクト目標を達成し、必要な要素成果物を生成するために、プロジェクトチームが実行する作業を、要素成果物を主体に階層的に要素分解したもの」
- プロジェクトを細かな作業に分解した構成図。
- 作業を細かく分解し、適切な作業順に並べることで構築する。

### WBS導入のメリット
- **スコープの明確化**：プロジェクトにおける、やらないことが明確になる。
- **作業の洗い出し**：実行すべき、やるべき作業が明確になり、漏れを防ぐ。
- **計画の具体化**：全体管理と作業計画が明確化され、計画の精度が向上する。
- **実行の基準**：プロジェクト実施時は、作成した WBS に則ってそのまま実行する。

### WBSの作り方・手順
1. **スコープを明確にする**
   - 全体から詳細へと切り分けるトップダウンアプローチを採用する。
   - ゴールから逆算して今やるべきことを決めるバックキャスティングの視点で考える。
2. **大きな作業をグルーピングする**
   - 共通性のある大きな単位で分類する。具体的な例として、データ収集、データ分析、可視化などが挙げられる。
3. **相互関連を整理する**
   - タスク間の論理的な前後関係である作業順序を考慮する。例えば、データ収集をしたあとで分析するという流れを作る。
4. **各グループの作業を洗い出す**
   - 漏れや重複をなくすため、もれなくダブりなくを意味するMECEの原則を意識して分解する。

## 第6回 プロジェクトマネジメント（2）／ガントチャート

### ガントチャートの定義
> 「プロジェクト管理や生産管理において、工程管理・スケジュール管理を行うために用いられる表」
- 縦軸に作業内容や担当者などのタスク、横軸に日付や時刻などの時間を配置する。
- 作業ごとの期間を横棒であるバーの長さで視覚的に表現する。
- 第5回で作成したWBSの各タスクを、時間軸上へとマッピングすることで作成する。

### ガントチャート導入のメリット
- **全体の進捗の可視化**：プロジェクト全体のスケジュール感や進み具合がひと目で把握できる。
- **依存関係の明確化**：どの作業が終わらないと次の作業に進めないのかという、先行タスクと後続タスクの関係が視覚的に分かる。
- **リソースの最適化**：誰が、いつ、どの作業を担当するのかが明確になり、負荷の偏りを防ぐ。

### プロジェクトの二面管理
実践的なプロジェクト運営においては、ガントチャートを全体と当日という2つの視点で切り分けて管理する。
1. **WBS全体シート**
   - プロジェクトの立ち上げとなる準備や企画から、本番、そして事後処理である片付けや評価に至るまでの長期的なすべての作業を網羅する。
2. **WBS当日シート**
   - 開催当日のオペレーション、タイムライン、進行、および想定されるトラブル対応などを、時間単位や分単位の時系列で詳細に管理する。

### プロジェクトの評価基準
- 作成したWBSおよびガントチャートに則って、実際にプロジェクトを過不足なく遂行できるかどうかが、計画自体の最大の評価基準となる。

## 第13回 CI/CDの実践

### GitHub Pages
- Githubにあるリモートリポジトリの内容をWebページとして公開できる。
- HP更改はCI/CDにより自動化される。
- Jekyllを使用することでMarkdownファイルを自動でHTML化できる。

### GitHub Pagesの設定手順
1. 課題用リポジトリ名をSettingsタブから`<user>.github.io`に修正する。
2. Settings→Pages→Build and deploymentのSourceを、deploy from a branchからGitHub Actionsに設定する。
3. GitHub Pages Jekyllのconfigureを実行し、デフォルトのままymlをコミットする（mainブランチでもよい）。
4. devブランチ内に公開用フォルダ（例：docs）を作成し、`jekyll new --skip-bundle .`でJekyllサイトを作成する。
5. docsディレクトリで`bundle install`を実行し、.gitignoreにGemfile.lockを追加する。

### _config.ymlの設定
- **title**：サイトのタイトルを変更する。
- **baseurl**：公開先のサブパス（例：`/docs`）を設定する。
- **url**：自分のアカウント名を使ったベースURL（`https://<user>.github.io`）を設定する。

### ページの追加とデプロイ
1. 新規Markdownファイルをdocs配下に移動し、layout・title・permalinkをYAMLで設定する。
2. index.markdownにそのファイルへのリンクを作成する。
3. mainブランチへプルリクエストを作成し、マージする。
4. 公開URL：`https://アカウント名.github.io/hello-world/docs/`
